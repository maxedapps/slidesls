# slidesls v2 — Ground-Up Rebuild into a Skill-Guided Slide Design System

**Date:** 2026-07-07
**Status:** Final plan (externally reviewed by Claude Opus 4.8; all major findings incorporated — see §11)
**Scope:** Phases 0–4, releasable per phase (0.6.0 → 1.0.0)

---

## 1. Summary

slidesls today is structurally excellent and visually impoverished: one visual noun (the 1px-bordered box), no loaded typefaces, no icon system, no imagery/diagram/chart vocabulary, a runtime in which slide transitions are physically impossible (`display:none` toggling), and guidance that is a defect-avoidance system with zero positive taste anchors. Agents therefore converge on the same slide: eyebrow + huge title + grid of half-empty boxes.

This plan rebuilds the product from the ground up into the target described in the vision document: **a skill-guided, registry-powered slide design system for agents** — polished building blocks (registry), discoverability (CLI), taste and process (skill), honesty (validation + visual QA).

Confirmed direction (user decisions):

- **Clean break.** Old primitives, templates, themes, and examples are deleted, not aliased. Copied decks keep working (shadcn model); only future authoring changes.
- **Vendored webfonts.** OFL variable woff2 files ship in the registry and are copied into decks (offline-capable, dependency-free).
- **New art-direction lineup.** Deliberately distant directions replace the five recolor themes.
- **Ship per phase.** Each phase lands as a releasable minor version with all gates green; 1.0.0 when the system is complete.
- **GSAP deferred** to post-1.0 (free since 3.13 but proprietary-licensed; must never be vendored as if MIT — see §3.6).

Core strategy: **fix the physics first, encode taste in defaults and lints rather than prose, build a design-review harness (including motion review) before growing the registry, and gate every phase on structured rendered review — not just green checks.** Structural gates (lints, scorecards) are _necessary, never sufficient_; the human taste gate is a bounded, rubric-driven protocol (§4.13).

## 3. Platform research findings (verified 2026-07)

1. **CSS `transition-behavior: allow-discrete` + `@starting-style`** — Baseline Newly Available since Aug 2024 (Chrome/Edge 117, Safari 17.5, Firefox 129). Entry animation needs `@starting-style` (beware: it also fires on first render — initial-load suppression required); on exit the `display:none` flip is delayed until the transition ends. Used for **child entrances/stagger**, where interruption doesn't matter.
2. **Web Animations API** (`element.animate()`) — Baseline Widely Available everywhere we target; offers `cancel()/finish()`, `Animation.finished`, `commitStyles()`. Used for **slide-level transitions**, where clean interruption under rapid navigation is the hard requirement.
3. **Same-document View Transitions API** — Baseline Newly Available since Oct 2025 (Firefox 144), but: page frozen during transition, snapshot overlay eats pointer events, duplicate `view-transition-name`s silently skip, and `prefers-reduced-motion` is _not_ respected automatically. **Rejected for v2 core**; revisit post-1.0 for shared-element section transitions.
4. **Fontsource variable fonts (latin `wght` woff2)** — Inter 47 KB, Fraunces 36 KB, Newsreader 57 KB, Space Grotesk 22 KB, Source Serif 4 50 KB, Libre Franklin 29 KB, JetBrains Mono 39 KB (IBM Plex Mono: no variable release). All SIL OFL 1.1: redistribution inside an MIT npm package and copying into user projects is permitted **provided each font family ships its own OFL.txt** (separate copyright holders — one license file per family, not per style).
5. **Lucide** — ISC license; embedding a curated SVG subset is fine with copyright notice retained. **External `<use href="sprite.svg#id">` is blocked on `file://`** (opaque-origin behavior; `<use>` is same-origin-only, no CORS opt-in). **Reliable offline pattern: inline `<symbol>` sprite in the document**, referenced via `<use href="#ls-i-name">`.
6. **GSAP** — free since 3.13 (Apr 2025) incl. former Club plugins, but under the proprietary GSAP Standard License (not MIT). May be used as an opt-in npm/CDN dependency in post-1.0 recipes; must not be vendored into the MIT package.

Codebase facts the plan builds on: registry items already carry rich schema-validated metadata (`schemas/registry-item.schema.json`, e.g. `registry/templates/split/registry-item.json` with `composition`, `agentLevel`, `useCases`); the visual QA collector already measures per-slide header/title/body rects, per-container fill ratios, grid child geometry, and min body font size (`src/validation/visual-qa-eval.mjs`); static lints live in `src/validation/design-lint.mjs`, measured analysis in `src/validation/visual-rhythm.mjs`; catalog/inspect flow through `src/registry/source.mjs`, `src/registry/catalog-doc.mjs`, `src/cli/agent-instructions.mjs`; init starter HTML in `src/deck/templates.mjs`; the release gate in `scripts/visual-gate.mjs` drives `agent-browser` when available.

## 4. Target architecture

### 4.1 Registry v2 layout

```
registry/
  core/           # reset, tokens, slide shell, motion core, runtime, icon helper  (mandatory)
  fonts/          # shared font pool: <family>/{*.woff2, OFL.txt}  (deduped across styles)
  styles/         # art directions: style.css + README (+ signature-font deps via fonts/)
  components/     # content vocabulary: surface, statement, stat, figure, list,
                  # quote, code, table, chart, flow, media, badge, progress, divider
  layouts/        # slide-body compositions with alignment guarantees + furniture
  archetypes/     # complete slide patterns: snippet(s) + content contract + variants
  motion/         # optional extra recipes beyond core defaults (emphasis, draw, typing)
  icons/          # curated Lucide subset: symbols + manifest + license
```

Renames vs v1: `presets/themes` → `styles`, `templates` → `archetypes`, `animations` → `motion`, `utilities/layout` → `layouts`. Old groups and items are **deleted** once replacements land (end of Phase 1 / Phase 2), never aliased. `registry.json`, `src/registry/source.mjs` group order (archetypes and styles surface first), and `catalog-doc.mjs` labels change accordingly. **During the transition releases (0.6.x), v2 authoring items carry `status: "preview"` and are hidden from default `catalog` output** so agents never see two vocabularies at once; v1 disappears and v2 unhides in the same release (0.7.0).

### 4.2 Tokens v2 (`core/base/tokens.css`)

Keep the `@layer tokens` + custom-property contract. Additions/changes:

- **Type roles with real scale:** `--ls-font-display/heading/body/label/code` (kept) plus used-in-anger display sizes (`--ls-text-hero` etc. actually consumed by components), per-density remaps kept.
- **Motion tokens (consumed, not decorative):** `--ls-transition-duration`, `--ls-transition-ease`, `--ls-transition-kind` (style's signature default), `--ls-enter-duration`, `--ls-enter-distance`, `--ls-stagger-step`.
- **Furniture tokens:** footer height, kicker treatment, page-number style hooks.
- **Texture hooks:** `--ls-slide-bg-image` kept, plus `--ls-slide-texture` layer and `--ls-deco-*` hooks styles can fill (grain, rules, scanlines, shapes).
- **Shape tokens:** radius scale kept; plus `--ls-border-width` so "sharp technical" vs "soft playful" is expressible.
- Every override-safe token documented via `authoring.cssVariables` so `inspect` surfaces the customization API (mechanism exists since 0.5.0).

### 4.3 Runtime v2 (`core/base/slide-runtime.js` + `core/base/motion.css`)

Rewrite around three ideas — **transitions by default, stagger by default, steps on demand** — with a fully specified interrupt model and choreography rule.

1. **Slide transitions — WAAPI-driven.** Slides remain absolutely stacked. On navigation the runtime runs `element.animate()` pairs (out on the leaving slide, in on the entering slide) parameterized by `data-ls-transition="fade | rise | slide | none"` on the deck (author attribute wins over the style's `--ls-transition-kind` token default; default `fade`). WAAPI is chosen for the slide layer specifically for **interruptibility**: on a new navigation during flight, the runtime calls `finish()` on both running animations (leaving slide reaches `display:none`+reset state via the finish handler, entering slide reaches identity), then starts the next pair — no stuck mid-`display` states, arbitrary key-spam safe. A `slidesls:transition` guard ensures exactly one pair runs at a time. Finish handlers own the `display`/`data-active` bookkeeping so CSS never races JS.
2. **Choreography rule (no double-motion sludge).** The slide-level transition and child stagger never both translate: with the default `fade` transition the _children_ provide movement (rise-in stagger); if the author picks `rise`/`slide` for the slide itself, child stagger automatically degrades to opacity-only (runtime sets `data-ls-stagger-mode="fade"`). This rule ships as core behavior, not guidance.
3. **Auto-stagger entrances — CSS-driven (`@starting-style` + `allow-discrete`).** On slide activation, the runtime assigns `--ls-enter-index` to **stagger units** found by this traversal, specified normatively: walk direct children of `.ls-slide__body` in DOM order; when a child is a layout/utility container (`.ls-layout-*`, `.ls-grid`, `.ls-stack`, `.ls-cluster`) or carries `[data-ls-stagger]`, descend one level and treat _its_ children as units (no deeper recursion; grid order = DOM order = row-major); cap at 12 units, remainder enters with the last index. Opt-outs: `data-ls-motion="none"` on deck or slide; `[data-step]` elements are excluded (steps own them). **Initial-load suppression:** entrance styles are keyed to a `data-ls-navigated` attribute the runtime sets only after the first user navigation — deck load renders slide 1 static (kills the `@starting-style` first-render flash).
4. **Steps.** Keep the `data-step` / `data-ls-reveal-sequence` contract and reveal-state machinery unchanged, but move the reveal CSS **into core** (`motion.css`) — stepping is no longer an opt-in registry item. The `motion/` group holds only _extra_ recipes (highlight, emphasis, draw-in, typing).
5. **Kill switches preserved everywhere:** `prefers-reduced-motion` (transition + stagger + steps become opacity-1ms), export mode (`?export=1`), and print collapse all motion — the current `reveal.css:27-45` behavior generalized. These are explicitly unit-tested.
6. **Furniture support:** runtime stamps `data-ls-slide-index`/`data-ls-slide-count` and fills `.ls-slide__footer [data-ls-page-number]` at init (works in deck, export, and print modes — CSS counters can't, because `display:none` slides don't generate counters).
7. Keep unchanged: hash state (`#slide=N&step=M`), keyboard map, scaling, `inert`/`aria-hidden`, `slidesls:ready` event. Lucide CDN auto-init is deleted (replaced by the sprite model).

### 4.4 Icons v2 (`registry/icons/` + `core/base/icons.css`)

- `scripts/icons-sync.mjs` (repo-side, dev-time): extracts a curated ~70-icon subset from `lucide-static` (devDependency) into `registry/icons/` as individual `<symbol>` files plus `manifest.json` and the ISC notice. Curation list lives in the script (checked in).
- **Delivery: inline sprite** (research §3.5 rules out external sprites on `file://`). `slidesls init` injects a hidden `<svg class="ls-sprite" aria-hidden="true">` after `<body>` with a starter symbol set; `slidesls icons sync --dir <deck>` scans deck HTML for `<use href="#ls-i-*">` references and rewrites the inline sprite to exactly the used set. `--add <name>` resolves from the curated local set first and **falls back to fetching the icon from the pinned lucide-static version on npm (unpkg/jsdelivr) when online**, so the 70-icon curation is a convenience tier, not a hard ceiling; offline without a local match fails with a clear message.
- Usage: `<svg class="ls-icon"><use href="#ls-i-zap"/></svg>`; `ls-icon`, `ls-icon-badge`, `ls-icon-mark` size/frame classes kept but re-specced per style.
- Validation: `unknown_icon` (referenced id missing from the deck sprite → error, with `icons sync` hint) and `emoji_icon` (emoji codepoint inside an icon slot → warning; suppressible per deck with a documented "intentionally emoji-styled" attribute, satisfying the vision's "no emoji icons unless explicitly stylistic").
- **One rule in guidance: sprite icons or nothing. No CDN Lucide, no glyph fallbacks.**

### 4.5 Typography v2 (fonts pipeline)

- **Shared font pool** `registry/fonts/<family>/` — each family dir holds latin variable woff2 + **that family's own OFL.txt** (separate copyright holders require per-family license files). Styles depend on families via `registryDependencies`, so `add`/`init` copies each family **once per deck** regardless of how many styles reference it (JetBrains Mono is the code face in all five directions; Inter is body in three — per-style dirs would ship them 5×/3×).
- `scripts/fonts-sync.mjs` (repo-side, dev-time): copies woff2 + OFL from pinned `@fontsource-variable/*` devDependencies into the pool. Families to vendor (≈8): Fraunces, Newsreader, Source Serif 4, Inter, Space Grotesk, Libre Franklin, JetBrains Mono, Bricolage Grotesque — ~320 KB once in the pool (final list per Appendix A).
- Each style's `style.css` contains its `@font-face` rules with **relative URLs** into the copied pool (works over `file://` and any server) and sets `--ls-font-*` roles on `:root[data-ls-style="<name>"]`.
- **The `data-ls-font` activation model is deleted** (root cause of D3). "Linked but inactive" becomes unrepresentable.
- Validation: `src/validation/assets.mjs` extended — every `@font-face src:` in copied CSS must resolve to an existing copied file (error: `font_asset_missing`); `pack:check` asserts a total package-size budget (< 5 MB).

### 4.6 Styles v2 (art directions)

A style = **tokens + fonts + texture + shape + furniture treatment + motion signature**, activated by a single `data-ls-style="<name>"` attribute on `<html>` set by `init --style`. Exactly one style per deck; `validate` errors on zero or multiple style links (`style_missing` / `style_conflict`).

Five directions are planned, chosen for maximum distance (full briefs in Appendix A): `editorial`, `terminal`, `gallery`, `boardroom`, `pop`. **Style count is decoupled from phase gates:** 0.7.0 ships the system with _at least three_ styles that pass the bar; any style that doesn't pass is deferred and ships as a 0.7.x increment when it does. A mediocre style never ships to make a date.

Acceptance bar for every style: the benchmark deck (§4.14) rendered in it must be unrecognizable as a sibling of the other styles at thumbnail size, and its motion signature must pass the motion review checklist (§4.13).

### 4.8 Layouts v2 + furniture

`.ls-layout--*` slide-body compositions with **alignment guarantees**, replacing bare grid-first composition:

- `--split` / `--columns` (2/3/4): columns are subgrid items spanning a **declared row skeleton** (`heading / body / footer` rows), so headings, bodies, and footers align across columns by construction — this kills D8's bottom-misalignment structurally. **Accepted tradeoff (explicit):** subgrid alignment dictates archetype markup shape; content must conform to the row skeleton. Each aligned layout also offers a `--free` variant that opts out of the skeleton for irregular content, trading alignment guarantees away deliberately rather than silently.
- `--statement`, `--dashboard` (fill grid for dense data), `--gallery`, `--band` (full-width horizontal flow), `--bleed` (content escapes slide padding on chosen sides).
- Each layout defines **where surplus space goes** (deliberate top-weighting or anchored furniture) instead of blanket `align-content:center`; `ls-stack/cluster/grid` remain as low-level utilities inside layout regions.
- **Slide furniture:** `.ls-slide__footer` (page number via runtime, running section kicker, deck title) styled per art direction. Furniture makes generous whitespace read as _designed_ (D8) and ships in every archetype snippet by default.

### 4.9 Archetypes v2 (with content contracts) and the image-sourcing ladder

Nine archetypes at launch (grow later behind the gallery gate): `title-hero`, `section`, `statement`, `big-stat`, `process-flow`, `comparison` (incl. before/after variant), `evidence` (quote + proof), `walkthrough` (code/terminal + annotations), `dashboard`. Backlog: `timeline`, `checkpoint`, `case-study`, `decision-matrix`, `agenda`.

Every archetype ships:

- `snippet.html` (+ variant snippets) marked `data-ls-archetype="<name>"`, using layouts/components/furniture, **with motion already wired** (steps where narrative, stagger otherwise) and **zero placeholder visuals**.
- **Content contract** in `registry-item.json` v2: slot counts and word ranges, e.g. process-flow `{ steps: {min:3,max:5}, stepTitle: {maxWords:4}, stepBody: {minWords:6,maxWords:16} }`. This is the structural fix for D11 — constrain the content, not just the CSS, and give agents what they're good at: writing to spec.
- `intent[]` (open/close/prove/compare/explain-process/teach/show-data/show-code/emphasize), style compatibility notes, recommended motion, icon guidance, `useWhen`/`avoidWhen`/`alternatives`/pairings (schema v2, extending the existing `composition` block).

**Image-sourcing ladder (decided here, not left to a lint).** Agents cannot produce raster images; banning placeholders without an alternative would make polished decks _harder_ to finish than v1. The sanctioned ladder for any figure slot, taught by the skill and encoded in archetype docs:

1. **Real asset** supplied by the user (screenshot, photo, logo) — always preferred; the skill's style-brief step asks for available assets up front.
2. **Authored diagram** using `flow`, `chart`, or hand-written SVG in a `figure` — for anything with inherent structure.
3. **`figure --abstract <style-variant>`** — the sanctioned intentional-graphic fallback (§4.7): token-colored, per-style generative compositions that are _designed to be shown_, not to apologize.
4. **Drop the slot**: switch to the archetype's no-figure variant (e.g. `title-hero --statement`) — every archetype with a figure slot MUST ship a no-figure variant.

Text-in-a-panel pretending to be a visual is never sanctioned (`placeholder_echo` lint). External image-generation integration is out of scope until post-1.0.

### 4.10 Validation & visual QA v2

Static (extends `design-lint.mjs` + new `src/validation/contract-lint.mjs`):

- `contract_slot_count`, `contract_copy_length` — archetype contract violations (advisory, precise hints).
- `archetype_monotony` — >50% of content slides share one archetype, or ≥3 consecutive repeats (D12).
- `placeholder_echo` — a `figure`/`surface` whose visible text (a) normalizes (lowercase, strip punctuation/whitespace) to < 60 chars **and** (b) has token-set Jaccard overlap ≥ 0.8 with the slide's eyebrow, badge, or title text, **or** matches a fixed placeholder-phrase list ("diagram", "visual", "image here", "placeholder", …). Advisory; thresholds are constants with fire/pass fixtures (D9).
- `icon_mix` / `emoji_icon` / `unknown_icon` (§4.4).
- `motion_absent` — no transition setting, no stagger, no steps, and deck not marked `data-ls-motion="none"` (which must be deliberate). Presence-only by design; motion _quality_ is verified by the motion review path (§4.13), not by lints.
- `style_missing` / `style_conflict` / `font_asset_missing` (§4.5, §4.6).
- `chart_a11y` — `chart`/`flow`/svg `figure` without `role="img"` + `aria-label`.

Measured (extends `visual-qa-eval.mjs` + `visual-rhythm.mjs`): collector additionally captures `data-ls-archetype`, icon usage, motion attributes, per-slide title font-size, and **computed foreground/background color pairs for text nodes** (nearest opaque ancestor); analyzer keeps header-rhythm/fill/legibility codes (recalibrated to v2 geometry), adds deck-level variety stats and `low_contrast` (WCAG: < 4.5:1 body text, < 3:1 display text — essential with five new color systems, especially phosphor-on-black and saturated-on-saturated directions).

`slidesls validate --report` prints a **deck scorecard**: per-slide archetype map, variety distribution, motion coverage, icon consistency, contrast summary, lint summary. **The scorecard is necessary, never sufficient**: it gates structure and honesty; taste is gated by the human review protocol (§4.13). Everything stays advisory-vs-error along the current philosophy: provable defects error; taste signatures warn loudly and are suppressible per slide (`data-ls-lint="off"`), so the system never blocks deliberate art direction.

### 4.11 Discovery CLI v2

- `catalog --intent <intent>`, `--style <name>` (compat filter), `--type archetype|component|layout|style|motion`; archetypes and styles first in group ordering; `--starter` curates the taste-safe subset.
- `inspect <item> --brief` (purpose, use/avoid, contract, motion, load tags — no snippet) and `--examples` (snippet + variants); default JSON keeps full metadata.
- Output prioritizes, in order: what it's for → use when / avoid when → content contract → density → style notes → recommended motion → load tags → snippet.
- `agent-instructions.mjs` rewritten: the help block walks the skill workflow (§4.12) and includes explicit discovery recipes for **every** group — closing D6's animation-discovery hole.

### 4.12 Skill v2 (`skills/create-slides-with-slidesls/`, same name)

SKILL.md restructured from plumbing+prohibitions to **process + taste, lint-backed**:

1. **Style brief** (required first artifact): audience, mood, pick one style (or custom-token direction), icon stance, motion level, density default, **available image assets** (feeds the §4.9 ladder).
2. **Deck rhythm plan**: list every slide as `archetype + one-line intent` _before writing HTML_; self-check variety (the monotony lint's thresholds are stated here).
3. **Build loop**: per slide — `inspect <archetype> --brief`, paste, write copy **to the contract**, customize via documented tokens/variants only.
4. **Motion pass**: defaults are free; add steps only where sequence carries meaning; verify motion by stepping the live preview (ArrowRight), not export screenshots.
5. **QA loop**: `validate` → fix errors → `visual-qa` → per-slide screenshots → judge against the positive checklist (focal point per slide? scale contrast? consecutive slides differ?).
6. **Done criteria**: scorecard clean (or deviations deliberately suppressed and explained) **and** rendered review performed — the skill states explicitly that a clean scorecard alone is not "done".

Hard rules section maps 1:1 to lints (each rule cites its code). References rewritten: `style-directions.md`, `archetypes.md`, `motion.md`, `customization.md`, `qa.md`, regenerated `catalog.md`. Flexibility clause kept and strengthened: user-provided style direction always wins; suppression is legitimate when intentional.

### 4.13 Design-review harness: gallery + motion + bounded human protocol (built FIRST, Phase 0)

`slidesls gallery` (new command, repo/dev-oriented): generates `.gallery/` HTML pages rendering **every registry snippet × every style × densities**, then (via `agent-browser`) screenshots each combo into `.gallery-review/<style>/<item>.png` and collects geometry payloads. `scripts/visual-gate.mjs` extends over the gallery: any measured defect code in any combo fails the release gate.

**Motion verification path (mandatory — motion is the headline feature and single frames cannot see it):**

- The gallery includes a **motion page per style**: a scripted navigation sequence captured as **timed frame bursts** (agent-browser screenshots at t ≈ 0/120/300/600 ms after a navigation, and during a stagger/step sequence). Frames are archived alongside the stills.
- Automated checks on the burst: entering slide opacity strictly increases; leaving slide is gone by t=600; stagger units have distinct paint-in times; rapid-nav script (5 navigations in 500 ms) ends on the correct slide with no orphaned mid-transition state. These catch interrupt jank and double-motion collisions mechanically.
- A **motion review checklist** (fixed, versioned in `docs/`) is run per style at each phase gate: signature matches the brief (bouncy vs understated), no rise-on-rise sludge, steps read as narrative, reduced-motion actually static.

**Bounded human review protocol** (the actual taste gate — structured so a single reviewer can hold the line over hundreds of combos):

- Per merge: the item's own screenshots across available styles (≤ ~12 images).
- Per phase gate: a **sampled rubric review** — every _new_ item in full, plus a rotating sample (each archetype × 2 styles, different pairs each phase), presented as side-by-side pairs; ≤ 60 images per session; five fixed rubric questions (focal point? hierarchy? intentional whitespace? on-brief for the style? would you present this?); verdicts recorded in `.gallery-review/REVIEW.md`.
- **The browser render path is a hard requirement for release**: `pnpm visual:gate` _fails_ (not skips) when `agent-browser` is unavailable in the release flow (`pack:check`); the default contributor `pnpm check` stays browser-free. (Today's skip-with-notice behavior would let both the taste gate and transition verification silently no-op.)

### 4.14 Benchmark ("four decks" test)

`benchmarks/deck-brief.md`: one fixed, realistic deck brief (content, audience, 8–10 slides). At Phase 2+ gates, the deck is authored via the skill in four styles; screenshots reviewed side-by-side under the §4.13 protocol. Pass = a stranger would not identify them as siblings; every archetype used at least once across the four; scorecards clean; motion checklist passes per style. This is the acceptance test for D7/D12 — polished _and varied_, not one new monotony.

## 5. Strategy rationale and alternatives considered

- **Hybrid motion engine: WAAPI for slide transitions, CSS (`@starting-style` + `allow-discrete`) for child entrances.** The slide layer's hard requirement is _interruptibility_ under rapid navigation — WAAPI's `finish()`/`cancel()` semantics solve what discrete-`display` CSS transitions are notoriously bad at (stuck mid-flip states). Child entrances don't need interruption handling, so declarative CSS with stagger vars keeps them cheap and style-themable. _(Rejected: pure-CSS `allow-discrete` for slides — interrupt handling unspecifiable; rejected: View Transitions API — frozen page, manual reduced-motion, silent skips; recorded as post-1.0 enhancement; rejected: pure-WAAPI everything — loses CSS themability of entrances.)_
- **Inline icon sprite over external sprite/CDN.** External `<use>` fails on `file://`; CDN violates offline/dependency-free. Cost: an `icons sync` command and slightly larger HTML. _(Rejected: per-icon inline SVG pasted everywhere — unmaintainable; rejected: optional CDN Lucide — the root cause of D5.)_
- **Vendored OFL fonts in a shared pool over system stacks.** System-only capped the ceiling (D2) and made styles converge; the pool dedupes shared families (JetBrains Mono ×5, Inter ×3 under per-style dirs) and carries per-family OFL texts for compliance. _(Rejected: runtime Google-Fonts links — breaks offline; rejected: per-style font dirs — multiplicative duplication.)_
- **Contracts + lints + defaults over prose rules.** Proven in-repo: the skill already said "animate by default" and 2/18 decks did. Every taste rule ships as a default (costs nothing to follow) or a lint (visible when broken); prose only explains _why_. Motion _quality_ is the exception that lints can't reach — hence the timed-capture path and checklist (§4.13). _(Rejected: skill-first rewrite before materials — cheap but inert, the 0.5 failure mode.)_
- **Ground-up v2 with deletion over incremental evolution.** User-confirmed. The copyable model makes this safe for existing decks; carrying both vocabularies would poison catalog discovery and double the QA matrix — which is also why v2 items stay hidden (`status: "preview"`) until v1 is deleted. _(Rejected: aliases; rejected: additive-only.)_
- **Vertical slice sequencing.** Phase 0 proves the whole pipeline (fonts, motion, icons, gallery incl. motion capture) on **one draft style** before Phase 1 fans out — avoids discovering infra problems five times. Within Phase 1, **all style token-shells are drafted before components are built**, so components are reviewed against the full style matrix once, not rebuilt per style.
- **Gallery + motion harness before registry growth.** N items × styles × densities is hundreds of visual states; without automated render/motion review and a bounded human protocol, quality regresses silently and the single reviewer rubber-stamps. Also guards the repo's known gravity toward tractable plumbing work: the harness makes design work reviewable, hence schedulable.

### Deviations from plan

1. `pairings` field implemented via the existing `pairsWith` (reference-validated) instead of a duplicate field.
2. `@font-face` lives in per-family `font.css` (own link tag via registryDependencies) instead of duplicated in each style.css — avoids duplicate rules for shared families.
3. Runtime ships as classic `defer` script (not `type="module"`): module scripts are CORS-blocked over file:// in Chromium; validation accepts both forms.
4. Taste lints live in `src/validation/taste-lint.mjs` (not folded into design-lint.mjs); contract lint in `contract-lint.mjs` with an exported slot→selector table guarded by the `contract_slot_unmapped` registry error.
5. `motion_absent`: with motion default-on, implemented as surfacing the deck-wide kill switch as a decision (the plan's "no transition setting" precondition no longer exists).
6. Comparison `rows` is prose guidance (copyGuidance), not a contract slot: rows may be lists, stats, or layout text, so a machine count would lie (review #3 confirmed false positives).
7. playwright-core added as a second visual-gate driver (agent-browser unavailable in this environment); agent-browser remains the default.
8. Archetype READMEs do not embed per-style gallery screenshots (plan 2.1): the stills live in `.gallery-review/` from the gate instead; embedding is post-1.0 polish.
9. Phases were executed and released in one continuous run; CHANGELOG carries the 0.6.0→1.0.0 series and package.json lands at 1.0.0. The maintainer may publish per-tag or once.
10. `icons list`/`icons sync` operate without a registry item for icons (sprite is CLI-managed); discovery via SKILL.md + `icons list`.
