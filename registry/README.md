# Registry

The copyable slidesls design system. Items are framework-free, dependency-light, and owned by the deck after copying: `slidesls add` resolves `registryDependencies`, copies files, and updates the deck manifest.

## Groups

- `core/` — the mandatory foundation: `core/base` ships reset, tokens, slide shell, icon contracts, `motion.css`, and `slide-runtime.js`. Every deck copies it first.
- `layouts/` — slide-body compositions with alignment guarantees (`layouts/core`): subgrid column layouts with a heading/body/footer row skeleton, statement/band/dashboard/gallery compositions, region bleed, plus the low-level stack/cluster/grid utilities.
- `components/` — the content vocabulary: standalone content, data, media, technical, and visual primitives (`surface`, `statement`, `stat`, `figure`, `list`, `code`, `chart`, `flow`, `media`, `quote`, `table`, `badge`, `divider`, `progress`).
- `archetypes/` — complete slide patterns with content contracts and motion wired in (`title-hero`, `statement`, `section`, `big-stat`, `process-flow`, `comparison`, `evidence`, `walkthrough`, `dashboard`). Each ships snippet(s) marked `data-ls-archetype`, slot/word constraints in `registry-item.json`, and `intent[]` metadata for discovery.
- `styles/` — the five art directions (`editorial`, `terminal`, `gallery`, `boardroom`, `pop`): tokens + fonts + texture + shape + furniture + motion signature, activated by one `data-ls-style="<name>"` on `<html>`. Exactly one per deck; styles pull their font families in via `registryDependencies`.
- `fonts/` — the vendored variable webfont pool (eight families, SIL OFL 1.1, per-family `OFL.txt`), generated by `scripts/fonts-sync.mjs` from pinned `@fontsource-variable/*` packages. Never edit by hand.
- `icons/` — the curated Lucide subset (ISC) as individual `<symbol>` files plus `manifest.json`, generated by `scripts/icons-sync.mjs`. Delivered to decks as an inline sprite maintained by `slidesls icons sync`.
- `motion/` — optional motion recipes beyond the defaults. Slide transitions, staggered entrances, and step reveals are **core** behavior (`core/base/motion.css` + the runtime) and need no item from this group; it exists only for extras (emphasis, line highlight, draw-in, typing).

## Item anatomy

Each item directory holds `registry-item.json` (see [docs/registry-contract.md](../docs/registry-contract.md)), a `README.md` with usage/API notes, implementation files when the item has copied assets, and optional `snippets/*.html`. The root `registry.json` indexes metadata files.

## Discovery and copying

```sh
slidesls catalog --json                   # complete lightweight inventory
slidesls catalog --type style --json      # pick the deck's art direction
slidesls inspect archetypes/big-stat --json
slidesls add layouts/core components/stat --dir <deck> --dry-run --json
```

## Validation

Run `slidesls validate-registry` after changing metadata or registry files, and review visual changes gallery-first (`slidesls gallery` + `scripts/visual-gate.mjs`).
